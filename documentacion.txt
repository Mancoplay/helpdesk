CAPITULO 1 - DESCRIPCION GENERAL DEL SISTEMA

1.1 Descripcion del sistema
El sistema Helpdesk es una aplicacion web desarrollada para administrar solicitudes de soporte tecnico mediante tickets. Su proposito es centralizar los reportes de incidencias, asignarlos a empleados o departamentos responsables, dar seguimiento al estado de atencion y mantener comunicacion entre usuarios, empleados y administradores.

El sistema esta orientado al area de soporte o mesa de ayuda de la institucion. Puede ser utilizado por usuarios que reportan problemas, empleados que atienden solicitudes y administradores que gestionan usuarios, empleados, departamentos, reportes y configuraciones generales.

El problema principal que resuelve es la falta de control cuando las solicitudes se manejan de forma manual, por mensajes informales o sin trazabilidad. Con el sistema se registra cada ticket, su estado, historial de mensajes, responsable, calificacion de atencion y notificaciones.

1.2 Stack tecnologico
Lenguaje principal: PHP 8.2 o superior.
Framework backend: Laravel 12.
Frontend y vistas: Blade, Livewire, Volt y Flux.
Plantilla visual: AdminLTE 4 beta.
Base de datos: PostgreSQL.
Servidor de desarrollo: php artisan serve.
Servidor web local recomendado: XAMPP o entorno compatible con PHP 8.2.
Gestor de dependencias PHP: Composer.
Gestor de dependencias JavaScript: npm.
Compilador de assets: Vite 6.
CSS y componentes: Tailwind CSS 4, Bootstrap 5, Sass.
Iconos y fuentes: Font Awesome, Source Sans 3.
Tiempo real: Laravel Reverb, Laravel Echo, Pusher JS.
Roles y permisos: Spatie Laravel Permission.
Pruebas: PHPUnit 11.

1.3 Arquitectura general del sistema
El sistema usa la arquitectura MVC propia de Laravel.

Modelo: representa las entidades y reglas de datos. Ejemplos: User, Ticket, Departamento, AreaTrabajo, TicketMensaje, TicketRemoteSession y SystemSetting.

Vista: contiene las pantallas que ve el usuario. Se ubican en resources/views y estan organizadas por modulos como tickets, usuarios, empleados, departamentos, notificaciones y reportes.

Controlador: recibe las solicitudes HTTP, valida datos, ejecuta la logica principal y retorna vistas o respuestas. El controlador principal identificado es HomeController, junto con NotificationController y controladores de autenticacion.

Base de datos: PostgreSQL almacena usuarios, tickets, departamentos, roles, permisos, sesiones, notificaciones, mensajes, eventos y configuraciones.

Servidor: Laravel atiende las rutas web, Vite sirve los assets durante desarrollo y Reverb permite notificaciones en tiempo real.

CAPITULO 2 - ANALISIS DE REQUISITOS

2.1 Descripcion del contexto y necesidad
Antes del sistema, las solicitudes de soporte podian registrarse manualmente o por canales dispersos, lo cual dificultaba saber quien solicito ayuda, que problema tenia, quien estaba atendiendo el caso y si la solicitud ya habia sido finalizada.

La automatizacion se vuelve necesaria para ordenar el proceso, evitar perdida de informacion, reducir tiempos de respuesta, generar historial y permitir que los responsables supervisen el trabajo del equipo de soporte.

2.2 Requerimientos de instalacion y entorno
Sistema operativo: Windows, Linux o macOS con soporte para PHP y PostgreSQL.
PHP: version 8.2 o superior.
Node.js: version 20 o superior.
Composer: requerido para instalar dependencias PHP.
npm: requerido para instalar y compilar dependencias frontend.
Base de datos: PostgreSQL.
Extensiones PHP recomendadas: pdo_pgsql, openssl, mbstring, tokenizer, xml, ctype, json, bcmath y fileinfo.
Memoria minima recomendada: 2 GB para desarrollo local; 4 GB o mas para un servidor compartido con cola y tiempo real.
Navegadores compatibles: Google Chrome, Microsoft Edge, Mozilla Firefox y otros navegadores modernos.

3.4 Modelo entidad-relacion
Tablas principales:

Un usuario empleado puede atender muchos tickets.
Un departamento puede tener muchos tickets.
Un departamento puede tener muchos empleados mediante departamento_empleado.
Un ticket puede tener muchos eventos o mensajes.
Un ticket puede tener una calificacion de atencion.

3.5 Diagrama de estados
Estados principales del ticket:

en_proceso: el ticket fue tomado o asignado para ser atendido.
finalizado: el empleado marco el ticket como resuelto.
cerrado: el proceso queda cerrado definitivamente o sin mas intervencion.

Flujo esperado:
pendiente -> en_proceso -> finalizado -> cerrado.
Tambien puede existir actualizacion, reasignacion o solicitud de soporte remoto durante el estado en_proceso.

CAPITULO 4 - DESARROLLO E IMPLEMENTACION

APP_NAME: nombre de la aplicacion.
APP_ENV: ambiente de ejecucion, por ejemplo local o production.
APP_DEBUG: activa o desactiva mensajes detallados de error.
APP_URL: URL base del sistema.
APP_LOCALE: idioma principal de la aplicacion.
DB_CONNECTION: motor de base de datos, en este caso pgsql.
DB_HOST: direccion del servidor de base de datos.
DB_PORT: puerto de PostgreSQL, normalmente 5432.
DB_DATABASE: nombre de la base de datos.
DB_USERNAME: usuario de la base de datos.
DB_PASSWORD: contrasena de la base de datos.
SESSION_DRIVER: mecanismo de almacenamiento de sesiones.
CACHE_STORE: mecanismo de cache.
QUEUE_CONNECTION: mecanismo de colas.
BROADCAST_CONNECTION: conexion usada para eventos en tiempo real.
REVERB_APP_ID, REVERB_APP_KEY y REVERB_APP_SECRET: credenciales locales de Reverb.
HELPDESK_CHAT_ATTACHMENTS_DISK: disco donde se guardan adjuntos.
HELPDESK_CHAT_ATTACHMENTS_DIR: carpeta de adjuntos de mensajes.
HELPDESK_PENDING_TICKET_REMINDER_MINUTES: minutos para recordar tickets pendientes.
HELPDESK_NOTIFICATIONS_RETENTION_DAYS: dias de retencion de notificaciones.
MAIL_MAILER: canal de correo, por ejemplo log en desarrollo.

4.2 Instalacion paso a paso
1. Clonar o copiar el proyecto en el equipo.
2. Entrar a la carpeta del proyecto.
3. Instalar dependencias PHP:
   composer install
4. Instalar dependencias JavaScript:
   npm install
5. Crear el archivo .env desde .env.example.
6. Configurar conexion PostgreSQL en .env.
7. Crear la base de datos helpdesk en PostgreSQL.
8. Generar clave de aplicacion:
   php artisan key:generate
9. Ejecutar migraciones:
   php artisan migrate
10. Cargar datos iniciales:
   php artisan db:seed
11. Crear enlace de storage:
   php artisan storage:link
12. Limpiar cache:
   php artisan optimize:clear
13. Levantar el sistema completo:
   composer run dev
14. Abrir en el navegador:
   http://127.0.0.1:8000

4.3 Descripcion de modulos del sistema
Modulo de autenticacion:
Permite iniciar sesion, verificar correo si aplica y recuperar contrasena. Usa rutas generadas por Laravel UI y controladores de autenticacion.

Modulo dashboard:
Muestra informacion general del sistema a usuarios con permiso de ver dashboard.

Modulo tickets:
Permite crear, listar, ver, editar, atender, finalizar, calificar y eliminar tickets. Tambien incluye mensajes, adjuntos, datos en vivo y sesiones remotas.

Modulo usuarios:
Permite al administrador registrar, actualizar, revisar y controlar usuarios del sistema.

Modulo empleados:
Permite administrar empleados, revisar informacion y activar controles internos.

Modulo departamentos:
Permite crear y actualizar departamentos, administrar correo de notificaciones y controlar disponibilidad.

Modulo notificaciones:
Permite consultar notificaciones, abrirlas, ver resumen de no leidas y marcarlas como leidas.

Modulo reportes:
Permite consultar reportes relacionados con usuarios y posiblemente con actividad del sistema.

Modulo configuracion de cuenta:
Permite editar perfil, cambiar contrasena y ajustar apariencia.

4.5 Base de datos: scripts y estructura
La base de datos se construye mediante migraciones de Laravel. Las migraciones principales crean tablas de usuarios, sesiones, cache, jobs, permisos, departamentos, tickets, relaciones departamento-empleado, notificaciones, configuraciones, eventos de ticket y areas de trabajo.

Tabla users:
Almacena datos de usuario como nombre, correo, contrasena, verificacion, departamento, area de trabajo, puntuacion promedio y cantidad de puntuaciones.

Tabla tickets:
Almacena codigo unico, cliente, empleado asignado, departamento, asunto, descripcion, estado, fecha de cierre, datos de notificacion, calificacion y solicitud de asignacion.

Tabla departamentos:
Almacena nombre, descripcion y estado activo.

Tabla areas_trabajo:
Almacena nombre, descripcion y estado activo del area de trabajo.

Tabla departamento_empleado:
Relaciona usuarios empleados con departamentos. Evita duplicados mediante restriccion unica.

Tabla ticket_eventos:
Almacena eventos de ticket, mensajes, adjuntos, solicitudes remotas, estados y notas.

Tabla notifications:
Almacena notificaciones generadas por el sistema.

Tablas de permisos:
Gestionan roles, permisos y sus relaciones con usuarios mediante Spatie Laravel Permission.

Para crear la estructura se ejecuta:
php artisan migrate

Para cargar datos iniciales se ejecuta:
php artisan db:seed

Las pruebas funcionales deben validar que cada pantalla y accion cumpla el resultado esperado sin revisar el codigo interno. Se recomienda probar los flujos como usuario, empleado y administrador.

Resultados esperados:
El usuario puede crear tickets y consultar su estado.
El empleado puede atender y finalizar tickets asignados.
El administrador puede gestionar usuarios, empleados y departamentos.

Las notificaciones se muestran y pueden marcarse como leidas.

Flujo integrado 1: usuario crea ticket, empleado lo atiende, empleado responde, empleado finaliza, usuario califica.
Flujo integrado 2: administrador crea departamento, asigna empleado, usuario crea ticket para ese departamento y empleado lo visualiza.
Flujo integrado 3: ticket genera notificacion, usuario abre notificacion y el sistema la marca como leida.
Flujo integrado 4: empleado solicita sesion remota y se actualiza el estado de la solicitud.

5.4 Errores comunes y soluciones
Error 1: No conecta con PostgreSQL.
Causa: base de datos no existe, credenciales incorrectas o extension pdo_pgsql deshabilitada.
Solucion: crear la base, corregir .env, habilitar pdo_pgsql y ejecutar php artisan optimize:clear.

Error 2: No cargan estilos o JavaScript.
Causa: dependencias Node no instaladas o Vite no esta ejecutandose.
Solucion: ejecutar npm install y npm run dev.

Error 3: No se puede iniciar sesion.
Causa: no se ejecutaron seeders o credenciales incorrectas.
Solucion: ejecutar php artisan db:seed y usar credenciales de prueba.

Error 4: No funcionan notificaciones en tiempo real.
Causa: Reverb no esta iniciado o variables VITE_REVERB estan mal configuradas.
Solucion: ejecutar php artisan reverb:start o composer run dev y revisar .env.

Error 5: Archivos adjuntos no se muestran.
Causa: falta enlace publico de storage o ruta de adjuntos mal configurada.
Solucion: ejecutar php artisan storage:link y revisar variables HELPDESK_CHAT_ATTACHMENTS.

5.5 Seguridad del sistema
Autenticacion: las rutas internas requieren usuario autenticado.
Roles y permisos: se usa Spatie Laravel Permission para limitar accesos.
Proteccion CSRF: Laravel protege formularios contra solicitudes externas no autorizadas.
Validacion: los datos deben validarse antes de almacenarse.
SQL injection: Eloquent y el query builder usan consultas parametrizadas.
Hash de contrasenas: Laravel almacena contrasenas de forma segura mediante hashing.
Throttling: algunas rutas aplican limites de frecuencia para evitar abuso.
Soft deletes: los tickets pueden eliminarse logicamente sin perder historial inmediato.
Variables de entorno: credenciales y claves deben mantenerse fuera del repositorio publico.

4. El historial de mensajes, estados, notificaciones y calificaciones aporta trazabilidad al proceso de soporte.
